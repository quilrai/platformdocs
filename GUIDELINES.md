# Repository Guidelines

This repository is the customer-facing QuilrAI platform documentation site. Use these notes when a
coding agent needs to decide where to make documentation changes.

## Where To Add Content

### Platform documentation

Update `docs/platform-areas/*.md` when a change affects current product behavior, workflows,
settings, permissions, visibility, or customer-facing terminology.

- Use one file per platform area.
- Keep pages written as durable baseline documentation, not as announcements.
- Preserve existing URLs and filenames unless a redirect or migration is part of the task.
- Add or update related links when a workflow spans multiple platform areas.
- If adding a new platform area, add a new `docs/platform-areas/<slug>.md` file, update
  `docs/overview.md`, and choose a sidebar icon that exists in `src/utils/sidebarIcons.js`.

Use `docs/overview.md` for the top-level platform summary, reading path, and links to major
platform areas. Update it when a new first-class area is added or when the high-level product
positioning changes.

### Release notes

Update `docs/release-notes/index.md` for customer-facing release notes.

- Add newest releases at the top.
- Use date headings in this format: `## June 8, 2026`.
- Keep bullets customer-facing and outcome-oriented.
- Link the primary platform area when the note maps clearly to one.
- Group related bullets under the same date when they describe one shipped change.
- Do not include internal-only refactors, file moves, tests, or implementation details unless they
  change customer-visible behavior.

Use `docs/release-notes/guidelines.md` for detailed release-note writing rules and automation
guidance. That page is marked `draft: true`; keep it as internal guidance unless explicitly asked to
publish it.

### Navigation and site structure

Use `sidebars.js` only for top-level sidebar structure. Platform-area pages are loaded through the
autogenerated `docs/platform-areas` category, so most new platform-area pages do not need a
`sidebars.js` edit.

Use `_category_.json` files for category labels, generated-index descriptions, positions, and
category icons.

Use `docusaurus.config.js` for site-level settings such as URL, base URL, navbar, footer, plugins,
themes, redirects, and docs configuration. Do not change deployment URLs or routing unless the task
is explicitly about hosting or navigation.

Use `static/` for static assets copied directly into the build, including `CNAME`, `.nojekyll`,
images, `robots.txt`, and `llms.txt`.

Use `src/components/` for reusable MDX components and `src/theme/` for Docusaurus theme overrides.
Avoid adding custom components when normal Markdown is enough.

## Documentation Style

- Write for customer admins, security teams, governance teams, IT teams, and field teams.
- Prefer product-area names used by the existing docs.
- Lead with customer-visible behavior and operational impact.
- Avoid internal component names, source paths, API routes, and engineering implementation details.
- Use concise headings and scannable bullets.
- Keep platform docs as the source of truth for current behavior; keep release notes as a dated
  summary of what changed.

## Frontmatter Conventions

Platform-area pages generally use:

```md
---
sidebar_position: 1
sidebar_custom_props:
  icon: Activity
---
```

Use icons from `src/utils/sidebarIcons.js`. If a new icon is needed, import it from `lucide-react`
and add it to `iconMap`.

Use `hide_copy_dropdown: true` only for pages where the page-copy dropdown should be hidden, such as
the overview and release-notes index.

## Change Checklist

When documenting a customer-visible product change:

1. Update the relevant baseline page in `docs/platform-areas/`.
2. Add a dated release note to `docs/release-notes/index.md` when the change is new, improved,
   fixed, or otherwise worth announcing.
3. Update `docs/overview.md` if the change affects the top-level platform summary or reading path.
4. Update sidebar icons or category metadata only if navigation changed.
5. Run `npm run build` before finishing when Markdown, MDX, links, config, or components changed.

When only release-note wording changes, edit `docs/release-notes/index.md` and check
`docs/release-notes/guidelines.md` for voice and classification rules.
